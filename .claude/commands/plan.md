`.specs/requirements-*.md` 파일 중 가장 최근 것을 읽고, 구현 가능한 스펙(`.specs/{번호}-{슬러그}.md`)으로 변환하세요.

## 변환 규칙

1. 각 FR(기능 요구사항)을 **측정 가능한 인수 기준**으로 변환
2. 기술 스택을 CLAUDE.md 기준으로 선택 (선택 이유 포함)
3. 태스크를 30분~2시간 단위로 분해
4. 각 태스크에 검증 방법 명시 (어떤 테스트로 확인할지)
5. 의존성 그래프 (어떤 태스크가 어떤 태스크에 선행)
6. 비목표를 명확히 (요구사항에 없는 건 절대 만들지 말 것)

## 휴먼 게이트

스펙 초안 작성 후 사용자에게 보여주고, 다음을 확인:
- "이대로 구현에 들어가도 될까요?"
- 사용자가 OK 할 때까지 수정
- 승인 후 파일을 `.specs/{다음번호}-{슬러그}.md`로 저장
- 번호는 기존 스펙 파일 보고 다음 번호로 (예: 002, 003)

## 출력 파일 구조

```markdown
# 스펙: {기능명}

> 기반 요구사항: requirements-{슬러그}.md

## 목표 (한 문장)

## 인수 기준
- AC-1: {측정 가능한 조건}
- AC-2: ...

## 기술 결정
- 사용 기술: ... (이유: ...)
- 라이브러리: ... (이유: ...)
- DB 스키마 변경: ...

## 태스크 분해

### T1: {태스크명} (예상: 30분)
- 작업 내용
- 검증 방법
- 의존: 없음

### T2: ...
- 의존: T1

## 비목표

## 리스크 / 의존성

## 검증 시나리오
- 시나리오 1: ...
```

## 주의

- 인수 기준은 "유저가 X 하면 Y가 보인다" 같이 외부에서 검증 가능하게
- 기술 결정은 CLAUDE.md 스택을 벗어나지 말 것
- 태스크는 너무 크게 잡지 말 것 (2시간 넘으면 쪼개기)
- 보안 / 검증 / 에러 처리는 별도 태스크로 분리

## 시작

대상 요구사항 파일: $ARGUMENTS (없으면 `.specs/requirements-*.md` 중 최신)

먼저 해당 파일 읽고 분석 결과를 사용자에게 보여주세요.
